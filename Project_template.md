# Задание 1. Исследование моделей и инфраструктуры

## 1. Сравнение LLM-моделей
| Критерий                           | Локальные модели (Llama 3, Mistral)                                                                                              | Облачные (OpenAI / YandexGPT)                                                                              |
|------------------------------------|----------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------|
| Качество ответов                   | Высокое, особенно у Mistral Large 3 (256K контекст, мультимодальность). Llama 3 подходит для кода                                | Очень высокое. GPT-4o mini — быстрый и недорогой вариант для базовых RAG-задач                             |
| Скорость работы                    | С GPU и оптимизациями (vLLM) — низкая задержка (< 100 мс)                                                                        | Зависит от API (~200–500 мс)                                                                               |
| Cтоимость владения и использования | Высокие начальные затраты на GPU, но низкая стоимость инференса при масштабировании                                              | GPT-4o mini: $0.2 / 1M токенов (input), $0.80 / 1M токенов (output). Доступно, но непредсказуемо при росте |
| Удобство и простота развёртывания  | Высокая сложность. Требуется управление зависимостями, CUDA, библиотеками. Решение: контейнеризация (Docker) и менеджеры пакетов | Максимально просто. Регистрация, API-ключ                                                                  |

**Вывод:** Для корпоративного RAG-бота с акцентом на контроль данных, предсказуемые затраты и масштабирование рекомендуется использовать локальные LLM. 
Облачные решения подходят для прототипирования, из-за простоты развертывания, но для продукта с растущей нагрузкой экономически невыгодны и небезопасны 

## 2. Сравнение моделей эмбеддингов
| Критерий                           | Локальные (Sentence-Transformers)                                                                                                      | Облачные (OpenAI Embeddings)                                                                                |
|------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------|
| Модель                             | all-MiniLM-L6-v2                                                                                                                       | text-embedding-3-large                                                                                      |
| Скорость создания индекса          | Очень высокая.  индексирует ~80000 записей/сек на CPU. Для 18k файлов — менее секунды                                                  | Низкая–средняя. Зависит от сети и лимитов API                                                               |
| Качество поиска                    | Среднее. показывает точность ~0.82 на стандартных бенчмарках (MTEB). Подходит для простых запросов, но может терять глубокую семантику | Высокая точность ~0.90. Лучше для сложных, многозначных вопросов                                            |
| Стоимость владения и использования | Минимальная. Модель весит 80 МБ, работает на CPU. Нет затрат на GPU                                                                    | Платно ($0.13 / 1M токенов). Для 18k документов → ~$5–10 за первичную индексацию + ежемесячно за обновления |

**Вывод:** all-MiniLM-L6-v2 — идеальный выбор, если:
- основной язык документации — английский
- критически важны скорость и низкое потребление ресурсов (работа на CPU)
- допустимо небольшое снижение точности поиска (на ~10% по сравнению с облачными моделями)

Для обработки документов на различных языках, рекомендуется использовать мультиязычную модель, например distiluse-base-multilingual-cased.

## 3. Сравните векторные базы ChromaDB и FAISS:
| Критерий                                 | ChromaDB                                                                                                              | FAISS                                                                                                          |
|------------------------------------------|-----------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------|
| Скорость поиска и индексации             | QPS: 24.68 (100k vectors). Latency: 2.58 мс/запрос. Скорость индексации: ~91.59s (5,551 chunks)                       | QPS: 344 (100k vectors). Latency: 0.34 мс/запрос. Скорость индексации: ~72.4s (5,551 chunks)                   |
| Сложность внедрения и поддержки          | Готовая БД. Проста в использовании, есть персистентность, метаданные, легко интегрируется с LangChain                 | Low-levely библиотека, требуется дополнительная СУБД. Нет персистентности, метаданных, шардирования из коробки |
| Удобство в работе                        | Быстрое прототипирование, MVP. Простота использования для маленьких и средниз датасетов (до нескольких млн. векторов) | Фокус на тюнинг производительности. Возможность использовать GPU                                               |
| Стоимость владения (учёт инфраструктуры) | Меньше кода, быстрее вывод в прод, но чуть выше операционные расходы на железо                                        | Требует разработки инфраструктуры вокруг, разворачивание и поддержка отдельной СУБД                            |

**Вывод:** ChromaDB оптимальный выбор для MVP и промышленной эксплуатации в данной задаче. Она балансирует скорость, функциональность и удобство поддержки, позволяя команде быстро итеративно улучшать качество поиска. FAISS рекомендую для команды с сильным ML-инжинирингом, готовой строить инфраструктуру с нуля.

## 4. Итоговые рекомендации

#### Вариант 1: FAISS + all-MiniLM-L6-v2 + локальный LLM
Плюсы: 
- максимальная скорость индексации и поиска, только на CPU
- минимальные затраты на железо, не нужен GPU
- простота масштабирования

Минусы: 
- Высокая сложность поддержки FAISS (нет метаданных), необходима отдельная инфраструктура для БД
- низкое качество поиска для не-английских языков

#### Вариант 2: ChromaDB + all-MiniLM-L6-v2 + Локальный LLM
Плюсы: 
- простота внедрения и поддержки ChromaDB
- максимальная скорость индексации и поиска, только на CPU
- минимальные затраты на железо, не нужен GPU
- идеально для быстрого MVP

Минусы:
- низкое качество поиска для не-английских языков

#### Вариант 3: ChromaDB + мультиязычная модель (distiluse-base-multilingual-cased) + Локальный LLM
Плюсы: 
- Хорошая поддержка мультиязычности. Баланс качества и скорости. Работает на CPU.

Минусы: 
- Медленнее all-MiniLM-L6-v2

#### Вариант 4: Cloud-native (Pinecone + OpenAI Embeddings + GPT)
Плюсы: 
- Высочайшее качество для любых языков.

Минусы: 
- Высокие и непредсказуемые расходы.


### Окончательный выбор
Для компании с офисами в Хельсинки и Таллине, где сотрудники могут создавать документацию на финском, эстонском или русском, целесообразней использовать Вариант 3 с мультиязычной моделью.
ChromaDB будет оптимальным выбором для MVP, а использование локальной LLM даст предсказуемые затраты и масштабирование.
